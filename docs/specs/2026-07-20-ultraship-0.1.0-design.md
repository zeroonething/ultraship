# UltraShip 0.1.0 ‚Äî Design

**Date:** 2026-07-20
**Status:** Approved
**Target release:** UltraShip 0.1.0 ‚Äî Single-product release workflow
**Source of requirements:** UltraShip Product and Implementation Specification v0.1.0 (not committed to this repo)
**Upstream:** [obra/superpowers](https://github.com/obra/superpowers) v6.1.1, MIT

---

## 1. What 0.1.0 delivers

A user installs UltraShip into Claude Code, and can then take one vague product idea from
nothing to a completed, immutable, release-ready version:

```
/ultraship:brainstorm ‚Üí /ultraship:plan ‚Üí /ultraship:develop
                          ‚Üï /ultraship:iterate
                        ‚Üí /ultraship:complete
```

State lives in `.ultraship/` inside the user's own project repository. Every fact has one
canonical owner. A released version cannot be silently edited.

This is itself a Minimum Complete Release: one user, one product, one workflow, end to end.

**Out of scope for 0.1.0** (each has its own spec'd release): multi-product portfolios,
release bundles, deployment hooks, provider usage telemetry, Codex and OpenCode adapters.

## 2. Decisions

Decisions made during brainstorming, beyond what the product specification already fixed.

| ID | Decision | Reason |
|----|----------|--------|
| B-01 | Markdown skills plus a small Node CLI, not prompts-only and not a full framework | Skills handle judgement. The CLI handles what language models do unreliably: schema validation, SemVer math, reference resolution, immutability. |
| B-02 | Canonical state is YAML | Product vision, requirements, and limitations are prose. JSON mangles multi-line text and produces noisy diffs. Resolves spec ¬ß29 Q1. |
| B-03 | Copy upstream files into our own history; no upstream remote, no fork | Avoids inheriting hundreds of unrelated commits and upstream's release tooling. |
| B-04 | Inherit six superpowers skills plus the dispatcher; leave the planning and agent-orchestration skills behind | Upstream `brainstorming`, `writing-plans`, and `executing-plans` compete with `/ultraship:brainstorm` and `/ultraship:plan`, violating D-003. Parallel-agent skills conflict with ¬ß14.5 resource awareness. |
| B-05 | Inherited skill bodies stay verbatim; only frontmatter names and `superpowers:` cross-references change | Rewriting sound prose for branding is the cosmetic fork ¬ß21.3 warns against. |
| B-06 | Distribution is the Claude Code plugin marketplace | Plugin namespacing produces `/ultraship:brainstorm` with no extra machinery. Resolves spec ¬ß29 Q11. |
| B-07 | `develop` recommends iteration and waits for the user; it never invokes `iterate` silently | D-014 requires an approval source for every plan change. Resolves spec ¬ß29 Q3. |
| B-08 | `complete` may fix release-blocking defects directly, but routes scope and requirement changes to `iterate` | Already stated in spec ¬ß13.5. Resolves spec ¬ß29 Q4. |
| B-09 | Release evidence stores trimmed command output in-repo | No CI integration exists in 0.1.0. Resolves spec ¬ß29 Q2 for this release only. |

## 4. The CLI

Five subcommands. Every one is deterministic and cheap; none call a model.

| Command | Behaviour |
|---------|-----------|
| `ultraship init` | Scaffolds `.ultraship/` with `ultraship.yaml` and `workspace.yaml`. Refuses if the directory already exists. |
| `ultraship state` | Prints workspace state, active product, active version, blockers, and the allowed next transitions as JSON. |
| `ultraship validate` | Validates every canonical file against its schema, checks SemVer values, resolves references, and verifies released records against the lock file. Exit code 0 or 1. |
| `ultraship semver next <current> <bump>` | Prints the next version. `bump` is one of `major`, `minor`, `patch`, or a pre-release identifier. |
| `ultraship views` | Regenerates `.ultraship/views/*.md` from canonical state, each carrying a do-not-edit notice. |

`ultraship state` exists so skills learn the workspace situation from one cheap call rather
than reading the whole `.ultraship/` tree into context on every invocation. This is the
concrete form of ¬ß14.5's inference-efficient behaviour.

### Immutability

`complete` records the SHA-256 of each released `<version>.yaml` in `.ultraship/releases.lock`.
`validate` recomputes those hashes and fails on any mismatch. Editing a released record is
therefore detectable by a command rather than prevented only by instruction.

## 5. Schemas

Nine JSON Schema files, one per canonical file type:

`ultraship`, `workspace`, `product`, `roadmap`, `release`, `active`, `tasks`, `iteration`,
`checkpoint`.

`registry`, `relationships`, and `release-bundle` are deliberately absent ‚Äî they belong to
0.2.0's multi-product work.

`release.schema.json` covers both halves of a release file: the contract written at plan time
(¬ß10.8) and the completion record written at release time (¬ß13.9). A file with
`status: released` additionally requires `released_at`, `evidence`, `mode`, and `immutable: true`.

### Canonical ownership

Ownership follows spec ¬ß15.3 unchanged. Every schema sets `additionalProperties: false`, so a
fact can only be written into the one file whose schema declares it. Writing `vision` into
`roadmap.yaml`, for example, fails validation rather than quietly creating a second source of
truth.

## 6. Skill contract

Every UltraShip skill follows the same shape:

1. Run `ultraship state`.
2. Check the requested transition is legal. If not, reroute per ¬ß19.3 and stop.
3. Do the skill's actual work, interacting with the user where the specification requires approval.
4. Write canonical YAML.
5. Run `ultraship validate`. On failure, fix the state ‚Äî never proceed with invalid state.
6. Run `ultraship views`.
7. Report what changed and name the next command.

Skills read `shared/state-model.md` rather than restating transition rules, so the rules have
one owner in this repository too.

### Per-skill notes

**brainstorm** ‚Äî refuses to run a second time once `workspace.yaml` records `BRAINSTORMED`, and
directs the user to `iterate`. Writes `product.yaml` only after explicit user approval.

**plan** ‚Äî writes `roadmap.yaml` plus a fully specified `releases/<version>.yaml` for the first
version, and outcome-level entries for later ones. Rejects roadmaps whose versions are
horizontal layers (¬ß10.5) by requiring every version to state a `user_can` outcome.

**develop** ‚Äî assesses release fit, generates just-in-time tasks in `tasks.yaml`, implements
vertical slices, and checkpoints. Invokes `ultraship:test-driven-development` for
implementation work. On detecting that the plan no longer holds, it stops and recommends
`iterate` with its evidence (B-07).

**iterate** ‚Äî writes an iteration record, updates affected canonical files, and resolves the
version consequence. Never edits a released record; never lowers a completion gate.

**complete** ‚Äî builds the gate checklist from the release contract, runs validation narrowest
first, stores evidence, writes the release record, updates the lock file, and recommends the
next version. Invokes `ultraship:verification-before-completion` before claiming anything passed.
A failed gate returns the release to `develop` or `iterate` and records which gate failed.

## 7. Resource awareness in 0.1.0

`resource_profile` lives in `ultraship.yaml`. Every field is optional and defaults to null.

The release fit gate produces a qualitative assessment only ‚Äî `high`, `probable`, `uncertain`,
or `unlikely` ‚Äî with written reasons and cost drivers. Skills are explicitly forbidden from
reporting token counts, percentages, or costs that they did not measure. Unknown stays unknown
(¬ß18.4).

Provider telemetry, budget flags, and model routing belong to 0.3.0.

## 8. Testing

`node:test`, no framework, no fixtures library.

- **Schema tests** ‚Äî a valid and an invalid fixture per schema; assert `validate` exit codes and
  that the error names the offending file and field.
- **SemVer tests** ‚Äî bump math including pre-release transitions.
- **Immutability test** ‚Äî release a fixture version, mutate its YAML, assert `validate` fails.
- **State machine test** ‚Äî assert every legal transition from spec ¬ß8 is permitted and that
  illegal ones are refused, including the rule that a released version never returns to a
  mutable state.
- **End-to-end fixture** ‚Äî one workspace walked from `UNINITIALIZED` to `RELEASED`, including a
  rejected completion that returns to `develop`.

## 10. Build order

1. Repository scaffold, licensing, NOTICE, `.gitignore`.
2. Import and rebrand the six inherited skills plus the dispatcher and session hook.
3. Schemas.
4. CLI: `init`, `state`, `validate`, `semver`, `views`, plus the lock file.
5. Tests for the CLI and schemas.
6. `shared/` reference documents.
7. The five skills, in lifecycle order.
8. End-to-end fixture walk.
9. README, CHANGELOG, plugin manifest, marketplace entry.

The framework's own 0.1.0 is complete when the acceptance criteria in specification ¬ß23 hold.
